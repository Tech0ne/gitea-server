# Gitea stack (Compose) — Gitea + Postgres + auto-registering runners

A self-contained Gitea setup where you edit **one config file**, run **one
command**, and get a working instance with its database in a separate container
and one (or more) Actions runners that **register themselves** — no clicking
around the admin UI to copy a token.

```
gitea-stack/
├── configs/
│   ├── stack.env        <- the ONE file you edit
│   └── runner.yaml      <- optional runner tuning (capacity, etc.)
├── docker-compose.yml
├── start.sh   stop.sh   Makefile
└── examples/demo-workflow.yaml
```

## Quick start

1. Edit `configs/stack.env` — at minimum change the passwords. Leave
   `RUNNER_REGISTRATION_TOKEN` on its `CHANGE_ME...` default and it will be
   generated for you.
2. Launch:
   ```bash
   ./start.sh          # or:  make up
   ```
3. Open `http://localhost:3000`, log in with `GITEA_ADMIN_USER` /
   `GITEA_ADMIN_PASSWORD`.
4. Check **Site Administration → Actions → Runners** — your runner is already
   there and online.

Stop it with `./stop.sh` (data is kept). Wipe everything with `make destroy`.

## How the auto-registration works

The usual pain with Gitea runners is that you must start the server, open the
admin UI, copy a registration token, and only then start the runner. This stack
skips that using a feature added in **Gitea 1.23**:

- The **Gitea server** is given `GITEA_RUNNER_REGISTRATION_TOKEN` (a reusable,
  global token). On startup Gitea registers it as a valid token.
- Each **runner** is given the *same* token plus `GITEA_INSTANCE_URL`. The
  runner image's entrypoint registers itself on first boot and then starts
  fetching jobs.
- `depends_on: condition: service_healthy` makes the runner wait for Gitea's
  health check to pass first, so the very first boot connects cleanly.

`start.sh` generates a proper 40-character token the first time (Gitea requires
exactly 40 chars) and writes it back into `stack.env`. Both sides read it from
there, so they always match.

## Running more than one runner

Set the count in `configs/stack.env`:

```
RUNNER_REPLICAS=3
```

then `./start.sh`. Each replica gets a distinct name (its container hostname)
and registers separately. They keep their registration across restarts; they
re-register if you recreate them (harmless, since the token is reusable).

## Choosing what jobs run in

`RUNNER_LABELS` in `stack.env` maps a label to the Docker image a job runs in:

```
RUNNER_LABELS=ubuntu-latest:docker://node:20-bookworm,ubuntu-22.04:docker://node:20-bookworm
```

A workflow's `runs-on: ubuntu-latest` then runs inside `node:20-bookworm`.
Jobs execute as sibling containers via the mounted Docker socket.

## Try it

Add `examples/demo-workflow.yaml` to a repo at `.gitea/workflows/demo.yaml`,
push, and watch it run under the repo's **Actions** tab. (Actions are enabled
globally here; you may still need to tick "Actions" in a repo's settings.)

## Notes & gotchas

- **Gitea image must be ≥ 1.23** (default here is `1.24`) or the global token is
  ignored and runners won't auto-register.
- **Mounting `/var/run/docker.sock`** gives runners full control of the host
  Docker daemon — fine for a homelab/CI box, but don't run untrusted workflows
  on it. For isolation, switch the runner image to a `*-dind-rootless` tag and
  drop the socket mount (see comments in the Gitea runner docs).
- **Web installer is skipped** (`INSTALL_LOCK=true`); the admin account is
  created by the one-shot `gitea-init` service instead.
- **`SECRET_KEY` / `INTERNAL_TOKEN`** are auto-generated by Gitea on first run
  and persisted in the `gitea_data` volume.

## Docker Swarm (`docker stack deploy`)

This file mostly works with Swarm (`RUNNER_REPLICAS` maps to native replicas),
but be aware:
- Swarm **ignores `depends_on` conditions**, so on a cold deploy a runner may
  log a few "connection refused" lines before Gitea is ready — it retries and
  recovers.
- Swarm doesn't read `--env-file` the same way; render first, e.g.
  `docker compose --env-file configs/stack.env config | docker stack deploy -c - gitea`.
- Bind mounts (`runner.yaml`, the Docker socket) must exist on whichever node
  the task lands on.
